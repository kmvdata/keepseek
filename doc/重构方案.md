结合当前源码，下面的方案力求构建一套更稳的 **Agent 运行时重构**：先稳定中断恢复，再降低探索成本，最后做压缩和真实 token 校准。

**重构目标**

把 KeepSeek 从“单次长跑，失败就停”改成“可重试、可续写、可降级总结、低成本检索”的 Agent runtime。核心边界仍保持现有架构：Provider 只编排，`AgentRunner` 管请求循环，工作区工具仍只读，写入仍走 DraftEdit。

**阶段一：运行中断治理**

1. 抽出 DeepSeek 请求层  
   新增 `src/agent/deepseek/client.ts`，把 [runner.ts](/Users/kermit/Projects/kmvdata/keepseek/src/agent/runner.ts:360) 里的 `fetch + SSE parse + 错误分类` 拆出去。  
   输出统一结果：`ok`、`finishReason`、`message`、`usage`、`hadPartialOutput`、`retryable`。

2. 安全 retry，只重试可重放请求  
   对 429、408、500、502、503、504、首 chunk 前网络失败做 retry。  
   SSE 已经输出 partial 后不要原样重放请求，而是交给“恢复/续写”逻辑。配置建议新增：
   - `keepseek.maxRequestRetries` 默认 2
   - `keepseek.requestRetryBaseMs` 默认 1000

3. `finish_reason=length` 做受限续写  
   只在这些条件满足时自动续写：没有 tool call、不是 DraftEdit 创建阶段、有可见正文、上下文预算仍够。  
   续写请求不带工具，追加一条 user 指令：“继续上一条回答，从截断处继续，不要重复”。最大 1-2 次，不建议默认 3 次。

4. 补齐预算触顶总结  
   当前 `tool_call_limit` 和 tool result budget 已经会注入总结指令，但 `maxToolIterations` 最后一跳仍可能直接返回上限文案。应把它也改成一次“无工具总结轮”，统一走 `getBudgetStopInstruction()`。

5. 修 Busy 输入行为  
   当前 busy submit 会直接 abort 并清空输入，见 [input/script.ts](/Users/kermit/Projects/kmvdata/keepseek/src/webview/input/script.ts:68)。建议改成：  
   - 点击停止按钮才 abort。  
   - 运行中键盘提交不 abort、不清空输入，只提示“当前任务运行中”。  
   - 可选 P1：支持排队一个 next prompt。

**阶段二：低成本代码探索**

1. 新增 `keepseek_search_workspace`  
   放在 [workspaceTools.ts](/Users/kermit/Projects/kmvdata/keepseek/src/agent/tools/workspaceTools.ts:12)，schema 放 [protocol.ts](/Users/kermit/Projects/kmvdata/keepseek/src/agent/protocol.ts:130)，路由放 [runner.ts](/Users/kermit/Projects/kmvdata/keepseek/src/agent/runner.ts:537)。  
   优先用 VS Code `workspace.findTextInFiles`，返回：路径、行号、匹配行、前后 2 行、truncated、limit。不要一开始引入额外 ripgrep 包。

2. 新增 `keepseek_read_workspace_file_range`  
   参数：`path`、`startLine`、`endLine`、`maxBytes?`。  
   返回：`content`、`startLine`、`endLine`、`totalLines`、`truncated`。  
   系统 prompt 也要改：先 search/list，再 range read，只有必要时读全文。

3. 改造 `read_workspace_file` 返回策略  
   保留全文读取工具，但对大文件返回“建议使用 range read”的结构化错误；小文件继续全文返回。这样能让模型自然转向片段读取。

**阶段三：上下文预算与压缩**

1. 引入工具结果分级保留  
   在 `AgentRunner` 内部维护 `toolResultLedger`，每个工具结果记录：工具名、路径、行区间、token 估算、是否可压缩。  
   大型 read/search 结果进入 messages 前先做 shaping：长行截断、最大结果数、最大字符数。

2. 第一版压缩不用模型  
   先做确定性压缩：  
   - read 文件结果压成：路径、语言、行数、读取范围、首尾摘要、关键匹配行。  
   - search 结果压成：查询词、命中文件列表、每文件命中数、前 N 条代表结果。  
   模型摘要放 P2，不要第一刀就引入额外模型调用。

3. Session Summary 后置  
   当前历史只取最近 24 条，见 [protocol.ts](/Users/kermit/Projects/kmvdata/keepseek/src/agent/protocol.ts:29)。可以新增 `summary` 字段到 `ChatSession`，但建议等 search/range 上线后再做。否则 summary 会掩盖工具低效的根因。

4. token 估算改进  
   先不要急着引入 `tiktoken`。更实用的是支持 DeepSeek streaming usage：请求体加 `stream_options.include_usage`，parser 记录真实 usage，再用真实 usage 校准当前 [tokenEstimate.ts](/Users/kermit/Projects/kmvdata/keepseek/src/agent/tokenEstimate.ts:1)。

**推荐落地顺序**

P0：DeepSeek client 抽层、retry、Busy submit 修复、`maxToolIterations` 总结轮。  
P1：`search_workspace`、`read_workspace_file_range`、系统 prompt 工具策略更新。  
P2：工具结果 ledger、确定性压缩、真实 usage 采集。  
P3：Session summary、模型辅助摘要、排队 prompt。

这套方案的重点是：不要先靠“自动续写”硬扛复杂任务，而是先让 Agent 少读、准读、可恢复。这样后续再做压缩和续航，收益会更扎实。
